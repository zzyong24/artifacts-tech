# /gen-script — 生成教学口播稿

根据用户提供的内容，生成一篇符合规范的教学短视频口播稿，并保存到指定作品目录。

## 使用方式

```
/gen-script <作品目录名> <内容来源>
```

**参数说明：**
- `<作品目录名>`：作品文件夹名，格式 `NN-主题简写`，如 `02-llm-context`
- `<内容来源>`：可以是文件路径、一段文字观点，或不填（进入交互问诊）

**示例：**
```
/gen-script 02-llm-context /path/to/article.md
/gen-script 02-llm-context 大模型的上下文窗口就像工作记忆，不是越大越好
/gen-script 02-llm-context
```

## 执行步骤

### Step 1：读取配置

读取 `.claude/artifacts.config.json`，获取：
- `prompts.script`：口播稿生成器 Prompt 模板路径
- `works_dir`：作品根目录
- `knowledge_base`：个人知识库根目录（可选，用于丰富个人视角）

如果配置文件不存在，提示用户：
> `.claude/artifacts.config.json` 不存在，请复制 `.claude/artifacts.config.example.json` 并填入本机路径。

### Step 2：读取 Prompt 模板

读取 `prompts.script` 路径下的 Markdown 文件，提取其中的**写作规范**和**结构要求**。

### Step 3：确定内容输入

- 若参数是文件路径 → 读取文件内容作为洞见素材
- 若参数是文字 → 直接作为洞见/观点输入
- 若无参数 → 询问用户：「你想讲的洞见/概念/观点是什么？」

### Step 4：从知识库取用背景素材（若 knowledge_base 已配置）

知识库是作者本人积累的笔记、文章、思考，是**个人声音**的来源，比任何通用资料都更有价值。

**取用策略**（知识库内容量大，必须精准检索，不能全量读取）：

1. **关键词提取**：从内容输入中提取 3-5 个核心关键词（主题词 + 领域词）
2. **精准搜索**：在 `knowledge_base` 目录下，用关键词搜索相关文件
   - 优先搜索 `writing/`、`reflections/`、`product/`、`AI/` 等子目录
   - 只读取与主题强相关的文件（标题或内容匹配关键词）
   - 一次最多取用 **3 篇**，不要贪多
3. **提炼个人视角**：从取用的内容中，找出：
   - 作者的亲身经历（「我在做 XX 的时候……」）
   - 作者的独特观点（与通用说法不同的地方）
   - 可用于「哲学延伸」的洞见或金句
4. **若搜索无结果**：跳过此步，直接进入 Step 5，不要强行填充

**重要原则**：
- 知识库素材是**增色**，不是主体。口播稿的核心洞见来自内容输入，知识库负责让「个人声音」更真实
- 不要把知识库内容直接搬进稿子，而是提炼成 1-2 个「真实经历切入点」或「个人观点补充」

### Step 5：生成口播稿（三阶段并行流程）

`{step}` 的位置依赖动画节点数量，而动画节点设计和逐字稿写作互不依赖——因此采用**并行生成 + 最终对齐**的流程，而非串行。

---

**5.1 先确定 Slide 骨架（串行，其他步骤的前置）**

基于 Step 4 的知识类型诊断结果，确定：
- 共几张 Slide，每张的主题边界
- 叙事结构（参考 `PEDAGOGY.md` §一）：反差型 / 问题-解答型 / 递进解包型 / …
- 每张 Slide 从 `source.md` 取用的核心素材

输出：**Slide 骨架表**（后续并行两侧的共同输入）

```
Slide 1：开场 — 反直觉断言
Slide 2：核心概念 — 类比 + 定义
Slide 3：机制拆解 — 步骤/流程
...
```

---

**5.2 并行生成（两侧同时展开，互不等待）**

| 侧 A：逐字稿内容 | 侧 B：动画节点规划 |
|-----------------|------------------|
| 按 Slide 骨架写每张的口播文字 | 按 Slide 骨架规划每张的动画步骤 |
| 全口语，短句为主，禁止套话 | 确定每张 Slide 有几个 `advance()` 节点（即 `slideSteps[n]` 的长度） |
| 字数由内容信息密度决定，不设硬性上限；内容丰富则合理拓宽，600–900 字是轻量内容的参考下限，不是目标 | 每个节点对应「展示什么内容」的简要描述 |
| 若有知识库素材，嵌入 1-2 处个人视角 | 输出格式：`Slide N → [节点1描述, 节点2描述, …]` |
| 结构：开场锚点 → 问题前置 → 类比降维 → 层层递进 → 哲学延伸（可选）→ 收尾行动 | |
| 金句引用：验证出处后自然嵌入，最多 1-2 句 | |

**侧 B 输出示例：**
```
Slide 2：知识的诅咒
  → 节点1：展示"敲节奏猜歌"实验场景
  → 节点2：揭示数据（预测50% vs 实际2.5%）
  → 节点3：定义框出现「知识的诅咒」
```

---

**5.3 对齐：插入 `{step}` 标记**

两侧完成后，逐张 Slide 对比侧 A（逐字稿）和侧 B（动画节点），在逐字稿中插入 `{step}`：

**规则：**
- 侧 B 有几个节点，侧 A 就对应插入几个 `{step}`
- `{step}` 插在「说完这句话后触发该节点动画」的句子末尾
- `{step}` 数量可少于动画节点数（节点可由下一句触发），但**不能多于**节点数
- 最后一个 `{step}` 之后的文字是该 Slide 的收尾口播

**对齐后格式（Slide 格式规范）：**

```markdown
### Slide 2：知识的诅咒

斯坦福大学做过一个实验，让一组人敲节奏猜歌名。 {step}

敲节奏的人预测猜对概率 50%，实际只有 2.5%。 {step}

这就叫「知识的诅咒」。 {step}

你懂得越多，就越难想起不懂是什么感觉。
```

**格式规范（提词器同步依赖）：**
- **必须用 `###`（三级标题）**，不能用 `##`
- 标题中**必须包含 `Slide N`**（N 为数字），提词器靠此识别锚点
- 每个 Slide 之间用 `---` 分隔
- 错误示范：`## Slide 1` / `**Slide 1**` / `# S1` 均无法被提词器识别

---

**⚠️ 生成后必须用 SUCCESs 六原则逐条校验（每条不通过则修改）：**

| 原则 | 含义 | 校验问题 | 不通过时的修法 |
|------|------|----------|----------------|
| **S — Simple（简单）** | 一期只讲一个核心洞见 | 这期内容的核心观点能用一句话说完吗？ | 删掉所有不服务于这一句话的内容 |
| **U — Unexpected（意外）** | 开场必须打破预期 | 开头的第一句话，观众能猜到后面要说什么吗？ | 改成反直觉的断言、反常识数据、或奇怪的问题 |
| **C — Concrete（具体）** | 用可感知的细节，不用抽象词汇 | 稿子里有没有「数字 / 场景 / 可以用感官感知的细节」？ | 把每一个抽象表达替换成一个具体例子或数字 |
| **C — Credible（可信）** | 来源可信 + 经历真实 | 这期内容的可信度来自哪里？数据来源？亲身经历？ | 加一个真实的实验/数据/亲身经历，或者明确承认"这是我的判断" |
| **E — Emotional（情感）** | 触动「这说的就是我」 | 观众会在哪一刻感到「这说的是我」或「哇这我没想到」？ | 加一个痛点场景描述，或者一个意外结论的情绪时刻 |
| **S — Story（故事）** | 最黏的内容是故事 | 稿子里有没有一个完整的「某人 + 某情境 + 某结果」的微故事？ | 把最核心的例子改写成 2-3 句的微故事形式 |

**重要提示：**
- SUCCESs 不是装饰，是强制校验。生成完稿后，必须逐条过一遍，有不通过的直接修改，不用告诉用户"我根据XXX修改了"，直接给出修改后的版本
- **Simple 是第一优先**：一期内容只有一个核心洞见。如果内容在 S 上不通过，整篇要重构

### Step 6：保存文件

保存到 `<works_dir>/<作品目录名>/script.md`。

如果目录不存在，自动创建，并同时创建一个空的 `README.md`（只写标题占位）。

**⚠️ 三文件规范：同时保存三份文件**

每次生成口播稿，必须同时保存以下三个文件：

| 文件 | 用途 | 内容 |
|------|------|------|
| `source.md` | 动画内容源（`/gen-slides` 的主要输入） | 原始完整文稿——用户传入的文章、笔记或完整素材原文，原样保存，不做删减 |
| `script.md` | 提词器加载（演讲时看） | 提纲稿（每行一个思想节点，以 `→` 开头）或逐字稿，根据用户偏好 |
| `script-full.md` | 逐字稿归档 | 完整逐字稿，保留备查 |

**`source.md` 保存规则：**
- 若内容来源是文件路径 → 将该文件内容**原样复制**到 `source.md`
- 若内容来源是用户直接输入的文字 → 将输入内容**原样保存**到 `source.md`
- 若内容来源是空（交互问诊模式）→ 将用户在问诊中提供的完整素材和背景信息整理后保存到 `source.md`
- `source.md` **只做保存，不做压缩**。口播稿是提炼产物，`source.md` 是信息全量，二者不可混同

**默认行为**：生成完整逐字稿后：
1. 将原始内容输入保存为 `source.md`
2. 将逐字稿保存为 `script-full.md`
3. 将逐字稿同内容也保存一份为 `script.md`（用户可以之后自行改为提纲版）

如果用户明确要求提纲稿，则：
1. 将原始内容输入保存为 `source.md`
2. 先生成完整逐字稿 → 保存为 `script-full.md`
3. 再提炼为提纲稿（每行关键词 + `→` 前缀）→ 保存为 `script.md`

**提纲稿格式示例：**
```markdown
### Slide 2：知识的诅咒

→ 斯坦福实验：敲节奏猜歌名 {step}
→ 敲的人预测 50% 猜对，实际 2.5% {step}
→ 知识的诅咒：你懂了就忘了不懂是什么感觉 {step}
→ 这就是为什么专家讲不清楚 {step}

---
```

### Step 7：输出确认

告知用户：
- 文件保存路径
- 口播稿共几张 Slide、预计时长
- 本次是否取用了知识库素材（取用了哪几篇）
- 下一步可执行 `/gen-slides <作品目录名>` 生成动画

**SUCCESs 自评报告（必须输出）：**

用下表格式输出本次稿件的 SUCCESs 评分，让用户一眼看到内容强度分布：

```
📊 SUCCESs 自评
S — Simple    ★★★★★  核心洞见：「[一句话]」
U — Unexpected ★★★★☆  开场：[开场第一句 or 反直觉设计]
C — Concrete   ★★★★★  最强具体点：[稿中最具体的细节/数据]
C — Credible   ★★★☆☆  可信来源：[实验/数据/经历 or 待补充]
E — Emotional  ★★★★☆  情感触点：[哪一句最能触动「这说的是我」]
S — Story      ★★★☆☆  故事性：[有/无，微故事位置]

最弱项：[C-Credible / S-Story 等] → 建议：[一句改进建议]
```

评分标准：
- ★★★★★ 非常强，无需修改
- ★★★★☆ 好，有小改进空间
- ★★★☆☆ 及格，建议演讲时补充
- ★★☆☆☆ 薄弱，建议修改后再录
- ★☆☆☆☆ 缺失，必须补上

## 附录：视频发布物料补充

### 封面主副标题 & 视频主副标题

口播稿生成完成后，必须同步输出以下发布物料，供封面制作和平台发布使用：

| 物料 | 数量 | 要求 |
|------|------|------|
| 封面主标题 | 2 个候选 | 6–10 字，冲击感强，可配合封面视觉；支持断行 |
| 封面副标题 | 2 个候选 | 10–18 字，一句话点出核心洞见，口语化 |
| 视频主标题 | 2 个候选 | 适合平台标题栏，含关键词，SEO 友好，≤ 30 字 |
| 视频副标题 | 2 个候选 | 补充说明或悬念延伸，≤ 25 字，与主标题形成张力 |

**写作原则：**
- 封面标题优先**视觉冲击**，字少力重，可用数字 / 反问 / 反常识断言
- 视频标题优先**搜索可见**，包含核心关键词，自然语言优先
- 主副标题之间形成「主→补充」或「结论→原因」的张力关系
- 禁止使用「揭秘」「震惊」「100%」等低质量标题党词汇

**输出格式示例：**

```
📌 封面标题候选
主标题 A：你学的越多，越不会教
主标题 B：专家的悲剧

副标题 A：— 知识的诅咒，如何打破？
副标题 B：— 学会了，却失去了理解别人的能力

📌 视频标题候选
主标题 A：为什么越厉害的人越讲不清楚？知识的诅咒真相
主标题 B：专家都有这个问题：懂了之后就忘了不懂是什么感觉

副标题 A：一个斯坦福实验揭示的认知陷阱
副标题 B：从「学会」到「教会」，你缺的是这一步
```
